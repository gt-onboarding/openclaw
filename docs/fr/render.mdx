---
title: Déployer sur Render
---

Déployez OpenClaw sur Render en utilisant l’Infrastructure as Code (IaC). Le Blueprint `render.yaml` inclus définit de manière déclarative l’intégralité de votre stack — service, disque, variables d’environnement — afin que vous puissiez déployer en un seul clic et versionner votre infrastructure aux côtés de votre code.

<div id="prerequisites">
  ## Prérequis
</div>

* Un [compte Render](https://render.com) (formule gratuite disponible)
* Une clé API de votre [fournisseur de modèles](/fr/providers) préféré

<div id="deploy-with-a-render-blueprint">
  ## Déployer avec un Blueprint Render
</div>

<a href="https://render.com/deploy?repo=https://github.com/openclaw/openclaw" target="_blank" rel="noreferrer">Déployer sur Render</a>

En cliquant sur ce lien, vous allez :

1. Créer un nouveau service Render à partir du Blueprint `render.yaml` situé à la racine de ce dépôt.
2. Être invité à définir la variable `SETUP_PASSWORD`
3. Construire l&#39;image Docker et la déployer

Une fois le déploiement terminé, l&#39;URL de votre service suit le schéma `https://<service-name>.onrender.com`.

<div id="understanding-the-blueprint">
  ## Comprendre le Blueprint
</div>

Les Blueprints Render sont des fichiers YAML qui définissent votre infrastructure. Le `render.yaml` dans ce dépôt configure tout ce qui est nécessaire pour exécuter OpenClaw :

```yaml
services:
  - type: web
    name: openclaw
    runtime: docker
    plan: starter
    healthCheckPath: /health
    envVars:
      - key: PORT
        value: "8080"
      - key: SETUP_PASSWORD
        sync: false          # prompts during deploy
      - key: OPENCLAW_STATE_DIR
        value: /data/.openclaw
      - key: OPENCLAW_WORKSPACE_DIR
        value: /data/workspace
      - key: OPENCLAW_GATEWAY_TOKEN
        generateValue: true  # génère automatiquement un jeton sécurisé
    disk:
      name: openclaw-data
      mountPath: /data
      sizeGB: 1
```

Principales fonctionnalités Blueprint utilisées :

| Feature               | Purpose                                                            |
| --------------------- | ------------------------------------------------------------------ |
| `runtime: docker`     | Construit l’image à partir du Dockerfile du dépôt                  |
| `healthCheckPath`     | Render surveille `/health` et redémarre les instances défaillantes |
| `sync: false`         | Invite à saisir une valeur au moment du déploiement (secrets)      |
| `generateValue: true` | Génère automatiquement une valeur cryptographiquement sécurisée    |
| `disk`                | Stockage persistant qui survit aux redéploiements                  |

<div id="choosing-a-plan">
  ## Choisir une offre
</div>

| Plan | Arrêt automatique | Disque | Idéal pour |
|------|-------------------|--------|------------|
| Free | Après 15 min d&#39;inactivité | Non disponible | Tests, démos |
| Starter | Jamais | 1 Go+ | Usage personnel, petites équipes |
| Standard+ | Jamais | 1 Go+ | Production, plusieurs canaux |

Le Blueprint utilise par défaut l’offre `starter`. Pour utiliser l’offre gratuite, modifiez `plan: free` dans le fichier `render.yaml` de votre fork (mais attention : sans disque persistant, la configuration est réinitialisée à chaque déploiement).

<div id="after-deployment">
  ## Après le déploiement
</div>

<div id="complete-the-setup-wizard">
  ### Terminer l&#39;assistant de configuration
</div>

1. Accédez à l&#39;adresse `https://<your-service>.onrender.com/setup`
2. Saisissez votre `SETUP_PASSWORD`
3. Sélectionnez un fournisseur de modèles et collez votre clé API
4. Le cas échéant, configurez les canaux de messagerie (Telegram, Discord, Slack)
5. Cliquez sur **Run setup**

<div id="access-the-control-ui">
  ### Accéder à l’interface Control UI
</div>

Le tableau de bord web est disponible à l’adresse `https://<your-service>.onrender.com/openclaw`.

<div id="render-dashboard-features">
  ## Fonctionnalités du tableau de bord Render
</div>

<div id="logs">
  ### Journaux
</div>

Affichez les journaux en temps réel dans **Dashboard → votre service → Logs**. Filtrez par :

* Journaux de build (création d’image Docker)
* Journaux de déploiement (démarrage du service)
* Journaux d’exécution (sortie de l’application)

<div id="shell-access">
  ### Accès au shell
</div>

Pour le débogage, ouvrez une session shell via **Dashboard → votre service → Shell**. Le disque persistant est monté sur `/data`.

<div id="environment-variables">
  ### Variables d’environnement
</div>

Modifiez les variables dans **Dashboard → votre service → Environment**. Toute modification déclenche un redéploiement automatique.

<div id="auto-deploy">
  ### Déploiement automatique
</div>

Si vous utilisez le dépôt OpenClaw d’origine, Render ne déploiera pas automatiquement votre instance OpenClaw. Pour la mettre à jour, effectuez une synchronisation manuelle du Blueprint depuis le tableau de bord.

<div id="custom-domain">
  ## Domaine personnalisé
</div>

1. Rendez-vous dans **Dashboard → your service → Settings → Custom Domains**
2. Ajoutez votre domaine
3. Configurez les enregistrements DNS comme indiqué (enregistrement CNAME pointant vers `*.onrender.com`)
4. Render provisionne automatiquement un certificat TLS

<div id="scaling">
  ## Mise à l’échelle
</div>

Render prend en charge la mise à l’échelle horizontale et verticale :

* **Verticale** : modifier l’offre pour obtenir plus de CPU/RAM
* **Horizontale** : augmenter le nombre d’instances (offre Standard et supérieures)

Pour OpenClaw, la mise à l’échelle verticale est généralement suffisante. La mise à l’échelle horizontale nécessite des sessions persistantes (« sticky sessions ») ou une gestion externe de l’état.

<div id="backups-and-migration">
  ## Sauvegardes et migration
</div>

Exportez votre configuration et votre espace de travail à tout moment :

```
https://<your-service>.onrender.com/setup/export
```

Télécharge une sauvegarde portable que vous pourrez restaurer sur n&#39;importe quel hôte OpenClaw.

<div id="troubleshooting">
  ## Dépannage
</div>

<div id="service-wont-start">
  ### Le service ne démarre pas
</div>

Vérifiez les journaux de déploiement dans le tableau de bord Render. Problèmes courants :

* `SETUP_PASSWORD` manquant — le Blueprint le demande, mais vérifiez qu’il est bien défini
* Incohérence de port — assurez-vous que `PORT=8080` correspond au port exposé dans le Dockerfile

<div id="slow-cold-starts-free-tier">
  ### Démarrages à froid lents (offre gratuite)
</div>

Les services de l&#39;offre gratuite sont arrêtés après 15 minutes d&#39;inactivité. La première requête après cet arrêt prend quelques secondes, le temps que le conteneur démarre. Passez à l&#39;offre Starter pour un service toujours en fonctionnement.

<div id="data-loss-after-redeploy">
  ### Perte de données après un redéploiement
</div>

Cela se produit avec l’offre gratuite (sans disque persistant). Passez à une offre payante ou exportez régulièrement votre configuration via `/setup/export`.

<div id="health-check-failures">
  ### Échecs de vérification d&#39;état
</div>

Render attend une réponse 200 de `/health` en moins de 30 secondes. Si les builds réussissent mais que les déploiements échouent, le service met peut-être trop de temps à démarrer. Vérifiez :

* Les journaux de build à la recherche d’erreurs
* Si le conteneur tourne en local avec `docker build && docker run`
